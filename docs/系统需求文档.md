# KB脚本执行框架系统需求文档

## 1. 简介

### 1.1 文档目的
本文档详细描述KB脚本执行框架的系统需求，旨在为开发团队提供技术实现指南，确保系统能够满足用户需求文档中定义的功能和性能要求。

### 1.2 术语定义
| 术语 | 解释 |
|------|------|
| KB | Knowledge Base，知识库，包含检查逻辑和规则 |
| 脚本 | 实现KB检查逻辑的可执行文件，支持多种脚本语言 |
| 框架 | 提供脚本执行环境、API和结果处理的基础平台 |
| API | 应用程序接口，框架提供给脚本使用的函数集合 |
| 归一化 | 将不同范围的数值转换到统一范围（如0-1）的过程 |
| 矩阵化 | 将多个脚本的执行结果组织成矩阵形式的数据结构 |
| 执行引擎 | 负责脚本执行的核心组件 |
| 结果处理器 | 负责处理和分析脚本执行结果的组件 |
| 脚本适配器 | 负责适配不同脚本语言的组件 |

## 2. 系统架构

### 2.1 总体架构
- **架构风格**：分层架构，包含执行层、适配层、处理层和接口层
- **核心组件**：
  - 执行引擎：负责脚本的执行和管理
  - 脚本适配器：负责适配不同脚本语言
  - API模块：提供通用API给脚本使用
  - 结果处理器：负责结果的处理和分析
  - 配置管理：负责系统配置的管理
  - 日志系统：负责系统日志的记录

## 3. 功能需求

### 3.1 CASE管理模块
- **功能描述**：负责CASE的定义、存储和查询
- **输入**：CASE配置文件、查询条件
- **输出**：CASE列表、CASE详情
- **详细需求**：
  - 支持从配置文件加载CASE定义
  - 支持按分类筛选CASE
  - 支持按标签筛选CASE
  - 支持按名称或描述搜索CASE
  - 提供CASE详情查询接口

### 3.2 场景管理模块
- **功能描述**：负责场景的定义、存储和执行
- **输入**：场景配置文件、执行请求
- **输出**：场景列表、场景详情、执行结果
- **详细需求**：
  - 支持从配置文件加载场景定义
  - 场景包含多个CASE的有序集合
  - 支持场景执行配置（并行/串行、超时等）
  - 提供场景列表查询接口
  - 提供场景详情查询接口

### 3.3 交互式模块
- **功能描述**：提供命令行交互式选择界面
- **输入**：用户键盘输入
- **输出**：选中的CASE列表、执行配置
- **详细需求**：
  - 支持多选CASE
  - 支持选择执行模式（串行/并行）
  - 支持配置执行参数
  - 提供友好的命令行界面

### 3.4 执行引擎
- **功能描述**：负责脚本的执行和管理
- **输入**：脚本文件路径、执行参数
- **输出**：脚本执行状态和结果
- **详细需求**：
  - 支持并行执行多个脚本
  - 支持脚本执行的超时控制
  - 支持脚本执行的异常捕获和处理
  - 提供脚本执行的状态查询接口
  - 支持脚本执行环境的隔离

### 3.5 脚本适配器
- **功能描述**：负责适配不同脚本语言
- **输入**：脚本文件、执行参数
- **输出**：脚本执行结果
- **详细需求**：
  - 支持bash脚本的执行（通过source方式组织多个脚本）
  - 支持Python脚本的执行
  - 提供统一的脚本执行接口
  - 负责脚本执行环境的准备
  - 处理不同脚本语言的API调用方式

### 3.6 API模块
- **功能描述**：提供通用API给脚本使用
- **输入**：API调用请求
- **输出**：API执行结果
- **详细需求**：
  - 提供log函数用于日志输出（对应bash的echo）
  - 提供result函数用于结果记录（对应bash的>>）
  - 提供step函数用于标记执行步骤
  - 提供不同脚本语言的API实现
  - 确保API接口的一致性和跨语言兼容性

### 3.7 结果处理器
- **功能描述**：负责处理和分析脚本执行结果
- **输入**：脚本执行的原始结果
- **输出**：标准化的结果和加权矩阵
- **详细需求**：
  - 解析脚本执行的原始结果
  - 生成标准化的JSON格式结果
  - 根据配置的权重系数进行加权计算
  - 对加权结果进行归一化处理
  - 生成多个脚本的执行结果矩阵
  - 提供结果导入后端系统的接口

### 3.8 配置管理
- **功能描述**：负责系统配置的管理
- **输入**：配置文件或配置API请求
- **输出**：系统配置
- **详细需求**：
  - 支持配置文件（如config.yaml）的加载
  - 支持环境变量配置
  - 支持脚本权重的配置
  - 支持API参数的配置
  - 提供配置验证和默认值处理

### 3.9 日志系统
- **功能描述**：负责系统日志的记录
- **输入**：日志信息
- **输出**：日志文件或日志流
- **详细需求**：
  - 支持不同级别的日志（DEBUG、INFO、WARN、ERROR）
  - 支持日志文件的轮转
  - 支持日志的结构化输出
  - 提供日志查询和分析接口

### 3.10 CASE初始化模块
- **功能描述**：负责创建标准化的CASE目录结构
- **输入**：CASE名称、脚本语言、目标目录
- **输出**：完整的CASE目录结构
- **详细需求**：
  - 支持指定CASE名称创建目录
  - 支持指定脚本语言（bash/python）
  - 自动生成配置文件（case.yaml）
  - 自动生成脚本模板文件
  - 支持交互式创建模式
  - 提供多种CASE模板选择

### 3.11 CASE目录结构规范
- **功能描述**：定义CASE目录的标准结构
- **输入**：无
- **输出**：CASE目录结构规范文档
- **详细需求**：
  - 每个CASE一个独立目录
  - 目录包含脚本文件和配置文件
  - 配置文件命名规范：case.yaml
  - 脚本文件命名规范：与目录同名
  - 支持多语言脚本文件

### 3.12 Web服务模块（P1）
- **功能描述**：提供HTTP服务和内嵌Web前端
- **输入**：HTTP请求
- **输出**：HTTP响应、静态资源
- **详细需求**：
  - 提供 `serve` 命令启动HTTP服务
  - 支持指定监听地址和端口
  - 支持Token认证
  - 内嵌Web前端静态资源
  - 提供RESTful API接口
  - 支持CORS跨域请求
  - CLI与Web服务功能隔离，互不影响

### 3.13 Token认证模块（P1）
- **功能描述**：提供Token认证保护Web服务
- **输入**：Token验证请求
- **输出**：验证结果、会话Cookie
- **详细需求**：
  - 支持通过命令行参数 `--token` 指定Token
  - 支持通过配置文件 `server.token` 指定Token
  - Token可选，不配置时不需要认证（仅本地使用）
  - Token验证成功后设置会话Cookie
  - 未认证用户访问时重定向到登录页面
  - 登录页面简洁，只需输入Token
  - 支持API Token验证（Header Authorization: Bearer token）
  - 启动时显示Token信息（Raw Token和Hashed Token）

### 3.13 执行历史模块（P1）
- **功能描述**：记录和管理执行历史
- **输入**：执行结果
- **输出**：历史记录列表、历史详情
- **详细需求**：
  - 每次执行生成一条历史记录
  - 历史记录存储在本地文件（JSON格式）
  - 支持查询历史记录列表
  - 支持查询单条历史详情
  - 支持删除历史记录
  - 历史记录文件命名：`history_{execution_id}.json`
  - 支持自动清除：当历史记录超过配置的最大条数时，自动删除最旧的记录
  - 默认最大条数：4294967296（2^32）
  - 支持手动清除全部历史记录

### 3.14 结果导出模块（P1）
- **功能描述**：支持导出执行结果
- **输入**：执行结果ID、导出格式
- **输出**：导出文件
- **详细需求**：
  - 支持JSON格式导出
  - 支持YAML格式导出
  - 导出结果符合标准化定义
  - 支持通过API和Web界面导出

## 4. 非功能需求

### 4.1 性能需求
- **响应时间**：脚本执行的启动时间不超过100ms
- **处理能力**：能够并行执行至少10个脚本
- **结果处理**：处理100个脚本的结果时间不超过1秒
- **资源使用**：内存使用不超过512MB，CPU使用率不超过50%

### 4.2 可靠性需求
- **错误处理**：能够处理脚本执行过程中的各种异常
- **容错能力**：即使部分脚本执行失败，系统仍能正常运行
- **稳定性**：系统能够连续运行72小时以上无故障
- **恢复能力**：系统故障后能够快速恢复并继续执行

### 4.3 可扩展性需求
- **模块化设计**：系统采用模块化设计，便于添加新功能
- **脚本语言支持**：支持通过适配器模式添加新的脚本语言
- **API扩展**：支持添加新的API函数
- **配置扩展**：支持通过配置文件扩展系统功能

### 4.4 安全性需求
- **执行隔离**：脚本执行应在隔离环境中进行，避免相互影响
- **权限控制**：限制脚本的执行权限，防止恶意操作
- **输入验证**：对脚本输入参数进行验证，防止注入攻击
- **数据安全**：保护执行结果和配置数据的安全

### 4.5 日志规范需求

系统应提供轻量、清晰的日志机制，便于问题排查。

#### 4.5.1 日志级别
| 级别 | 用途 | 示例 |
|------|------|------|
| DEBUG | 调试信息，开发阶段使用 | 变量值、执行路径 |
| INFO | 正常运行信息 | 步骤开始/结束、关键操作 |
| WARN | 警告信息，不影响主流程 | 配置缺失使用默认值、重试操作 |
| ERROR | 错误信息，影响当前操作 | 脚本执行失败、参数错误 |

#### 4.5.2 日志格式
```
[时间] [级别] [脚本名:步骤名] 消息内容
```

示例：
```
[2024-01-01 12:00:00] [INFO] [security_check:step1] 步骤开始执行
[2024-01-01 12:00:01] [ERROR] [security_check:step1] 文件权限检查失败: /etc/passwd
[2024-01-01 12:00:02] [INFO] [security_check:step1] 步骤执行完成, 耗时: 2s
```

#### 4.5.3 日志API设计

**Bash API**：
```bash
log_debug "message"    # DEBUG级别日志
log_info "message"     # INFO级别日志
log_warn "message"     # WARN级别日志
log_error "message"    # ERROR级别日志
```

**Python API**：
```python
kb.log_debug("message")  # DEBUG级别日志
kb.log_info("message")   # INFO级别日志
kb.log_warn("message")   # WARN级别日志
kb.log_error("message")  # ERROR级别日志
```

### 4.6 结果输出规范需求

系统应提供标准化的结果输出格式，确保结果的一致性和可读性。

#### 4.6.1 步骤结果规范

每个步骤的执行结果应包含以下字段：

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| name | string | 是 | 步骤名称 |
| status | enum | 是 | 状态：success/failure/warning/skipped |
| message | string | 否 | 执行消息 |
| output | string | 否 | 原始输出 |
| duration_ms | number | 否 | 执行耗时(毫秒) |

#### 4.6.2 脚本结果规范

脚本的整体执行结果应包含以下字段：

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| script_name | string | 是 | 脚本名称 |
| status | enum | 是 | 整体状态 |
| steps | array | 是 | 步骤结果列表 |
| score | number | 否 | 执行得分(0-1) |
| duration_ms | number | 否 | 总耗时(毫秒) |
| message | string | 否 | 整体消息 |

#### 4.6.3 结果API设计

**Bash API**：
```bash
# 步骤开始
step_start "step_name"

# 步骤结束
step_success "message"           # 成功
step_failure "message"           # 失败
step_warning "message"           # 警告
step_skip "message"              # 跳过

# 记录步骤输出
step_output "raw output content"

# 记录键值对结果
result "key" "value"
```

**Python API**：
```python
# 步骤开始
kb.step_start("step_name")

# 步骤结束
kb.step_success("message")       # 成功
kb.step_failure("message")       # 失败
kb.step_warning("message")       # 警告
kb.step_skip("message")          # 跳过

# 记录步骤输出
kb.step_output("raw output content")

# 记录键值对结果
kb.result("key", "value")
```

#### 4.6.4 输出格式规范

**JSON格式**：
```json
{
  "script_name": "security_check",
  "status": "success",
  "score": 0.95,
  "duration_ms": 2345,
  "message": "安全检查通过",
  "steps": [
    {
      "name": "check_permissions",
      "status": "success",
      "message": "权限检查通过",
      "output": "...",
      "duration_ms": 500
    }
  ]
}
```

**Table格式（CLI输出）**：
```
┌─────────────────────┬─────────┬──────────────────────┬──────────┐
│ Step                │ Status  │ Message              │ Duration │
├─────────────────────┼─────────┼──────────────────────┼──────────┤
│ check_permissions   │ SUCCESS │ 权限检查通过         │ 500ms    │
│ check_users         │ SUCCESS │ 用户检查通过         │ 300ms    │
│ check_config        │ WARNING │ 配置存在风险         │ 200ms    │
└─────────────────────┴─────────┴──────────────────────┴──────────┘
Script: security_check | Status: SUCCESS | Score: 0.95 | Duration: 2345ms
```

## 7. 部署需求

### 7.1 环境要求
- **操作系统**：Linux (Ubuntu 20.04+)、Windows (Windows 10+)
- **Go版本**：1.23+
- **依赖**：
  - bash (for bash脚本)
  - Python 3.8+ (for Python脚本)
  - 必要的Go依赖库

### 7.2 部署方式
- **二进制部署**：编译为可执行文件，直接运行
- **容器部署**：提供Dockerfile，支持容器化部署
- **服务部署**：作为系统服务运行，支持自动启动

### 7.3 配置文件
- **配置文件路径**：`configs/config.yaml`
- **配置示例**：
  ```yaml
  server:
    port: 8080
    host: 0.0.0.0
  execution:
    timeout: 300
    max_parallel: 10
  scripts:
    directory: "./scripts"
  logging:
    level: "info"
    file: "./logs/kb-runner.log"
  weights:
  ```

## 11. 执行历史存储设计（P1）

### 11.1 存储位置

- 默认目录：`./history/`
- 可通过配置文件指定

### 11.2 存储格式

每条历史记录存储为一个JSON文件：

```
history/
├── history_exec-1709876543210.json
├── history_exec-1709876543211.json
└── index.json                          # 历史索引文件
```

### 11.3 历史记录结构

```json
{
  "execution_id": "exec-1709876543210",
  "timestamp": "2024-03-08T10:30:43+08:00",
  "status": "success",
  "trigger": "web",
  "scenario": "daily_check",
  "cases": ["security_check", "performance_check"],
  "summary": {
    "total_scripts": 2,
    "success_count": 2,
    "failure_count": 0,
    "average_score": 0.95
  },
  "result": {
    "timestamp": "...",
    "execution_id": "...",
    "scripts": [...],
    "matrix": [...],
    "summary": {...}
  }
}
```

### 11.4 索引文件结构

```json
{
  "records": [
    {
      "execution_id": "exec-1709876543210",
      "timestamp": "2024-03-08T10:30:43+08:00",
      "status": "success",
      "trigger": "web",
      "scenario": "daily_check"
    }
  ]
}
```

### 11.5 历史记录配置

```yaml
history:
  enabled: true                    # 是否启用历史记录
  max_records: 4294967296          # 最大记录数（默认2^32）
  auto_cleanup: true               # 是否自动清理
  cleanup_threshold: 0.9           # 清理阈值（达到90%时触发）
```

### 11.6 清除策略

- **自动清除**：当历史记录数达到 `max_records * cleanup_threshold` 时，自动删除最旧的记录
- **手动清除**：
  - API: `DELETE /api/v1/history` 清除全部
  - API: `DELETE /api/v1/history/{id}` 清除单条
  - CLI: `kb-runner history clear` 清除全部

## 14. 风险/依赖备注

### 10.1 风险
- **脚本执行安全**：脚本可能包含恶意代码，需要加强执行隔离
- **API一致性**：不同脚本语言的API实现可能存在差异，需要统一接口设计
- **性能瓶颈**：并行执行多个脚本可能导致系统资源不足，需要优化资源管理
- **结果准确性**：权重配置的合理性可能影响结果的准确性，需要提供配置指导

### 10.2 依赖
- **脚本语言环境**：依赖bash、Python等脚本语言的运行环境
- **Go依赖库**：依赖必要的Go第三方库
- **后端系统**：依赖后端系统的API进行结果导入
- **操作系统**：依赖操作系统提供的进程管理和文件系统功能
