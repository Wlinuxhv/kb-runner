# KB脚本执行框架系统需求文档

## 1. 简介

### 1.1 文档目的
本文档详细描述KB脚本执行框架的系统需求，旨在为开发团队提供技术实现指南，确保系统能够满足用户需求文档中定义的功能和性能要求。

### 1.2 术语定义
| 术语 | 解释 |
|------|------|
| KB | Knowledge Base，知识库，包含检查逻辑和规则 |
| 脚本 | 实现KB检查逻辑的可执行文件，支持多种脚本语言 |
| 框架 | 提供脚本执行环境、API和结果处理的基础平台 |
| API | 应用程序接口，框架提供给脚本使用的函数集合 |
| 归一化 | 将不同范围的数值转换到统一范围（如0-1）的过程 |
| 矩阵化 | 将多个脚本的执行结果组织成矩阵形式的数据结构 |
| 执行引擎 | 负责脚本执行的核心组件 |
| 结果处理器 | 负责处理和分析脚本执行结果的组件 |
| 脚本适配器 | 负责适配不同脚本语言的组件 |

## 2. 系统架构

### 2.1 总体架构
- **架构风格**：分层架构，包含执行层、适配层、处理层和接口层
- **核心组件**：
  - 执行引擎：负责脚本的执行和管理
  - 脚本适配器：负责适配不同脚本语言
  - API模块：提供通用API给脚本使用
  - 结果处理器：负责结果的处理和分析
  - 配置管理：负责系统配置的管理
  - 日志系统：负责系统日志的记录

## 3. 功能需求

### 3.1 CASE管理模块
- **功能描述**：负责CASE的定义、存储和查询
- **输入**：CASE配置文件、查询条件
- **输出**：CASE列表、CASE详情
- **详细需求**：
  - 支持从配置文件加载CASE定义
  - 支持按分类筛选CASE
  - 支持按标签筛选CASE
  - 支持按名称或描述搜索CASE
  - 提供CASE详情查询接口

### 3.2 场景管理模块
- **功能描述**：负责场景的定义、存储和执行
- **输入**：场景配置文件、执行请求
- **输出**：场景列表、场景详情、执行结果
- **详细需求**：
  - 支持从配置文件加载场景定义
  - 场景包含多个CASE的有序集合
  - 支持场景执行配置（并行/串行、超时等）
  - 提供场景列表查询接口
  - 提供场景详情查询接口

### 3.3 交互式模块
- **功能描述**：提供命令行交互式选择界面
- **输入**：用户键盘输入
- **输出**：选中的CASE列表、执行配置
- **详细需求**：
  - 支持多选CASE
  - 支持选择执行模式（串行/并行）
  - 支持配置执行参数
  - 提供友好的命令行界面

### 3.4 执行引擎
- **功能描述**：负责脚本的执行和管理
- **输入**：脚本文件路径、执行参数
- **输出**：脚本执行状态和结果
- **详细需求**：
  - 支持并行执行多个脚本
  - 支持脚本执行的超时控制
  - 支持脚本执行的异常捕获和处理
  - 提供脚本执行的状态查询接口
  - 支持脚本执行环境的隔离

### 3.5 脚本适配器
- **功能描述**：负责适配不同脚本语言
- **输入**：脚本文件、执行参数
- **输出**：脚本执行结果
- **详细需求**：
  - 支持bash脚本的执行（通过source方式组织多个脚本）
  - 支持Python脚本的执行
  - 提供统一的脚本执行接口
  - 负责脚本执行环境的准备
  - 处理不同脚本语言的API调用方式

### 3.6 API模块
- **功能描述**：提供通用API给脚本使用
- **输入**：API调用请求
- **输出**：API执行结果
- **详细需求**：
  - 提供log函数用于日志输出（对应bash的echo）
  - 提供result函数用于结果记录（对应bash的>>）
  - 提供step函数用于标记执行步骤
  - 提供不同脚本语言的API实现
  - 确保API接口的一致性和跨语言兼容性

### 3.7 结果处理器
- **功能描述**：负责处理和分析脚本执行结果
- **输入**：脚本执行的原始结果
- **输出**：标准化的结果和加权矩阵
- **详细需求**：
  - 解析脚本执行的原始结果
  - 生成标准化的JSON格式结果
  - 根据配置的权重系数进行加权计算
  - 对加权结果进行归一化处理
  - 生成多个脚本的执行结果矩阵
  - 提供结果导入后端系统的接口

### 3.8 配置管理
- **功能描述**：负责系统配置的管理
- **输入**：配置文件或配置API请求
- **输出**：系统配置
- **详细需求**：
  - 支持配置文件（如config.yaml）的加载
  - 支持环境变量配置
  - 支持脚本权重的配置
  - 支持API参数的配置
  - 提供配置验证和默认值处理

### 3.9 日志系统
- **功能描述**：负责系统日志的记录
- **输入**：日志信息
- **输出**：日志文件或日志流
- **详细需求**：
  - 支持不同级别的日志（DEBUG、INFO、WARN、ERROR）
  - 支持日志文件的轮转
  - 支持日志的结构化输出
  - 提供日志查询和分析接口

### 3.10 CASE初始化模块
- **功能描述**：负责创建标准化的CASE目录结构
- **输入**：CASE名称、脚本语言、目标目录
- **输出**：完整的CASE目录结构
- **详细需求**：
  - 支持指定CASE名称创建目录
  - 支持指定脚本语言（bash/python）
  - 自动生成配置文件（case.yaml）
  - 自动生成脚本模板文件
  - 支持交互式创建模式
  - 提供多种CASE模板选择

### 3.11 CASE目录结构规范
- **功能描述**：定义CASE目录的标准结构
- **输入**：无
- **输出**：CASE目录结构规范文档
- **详细需求**：
  - 每个CASE一个独立目录
  - 目录包含脚本文件和配置文件
  - 配置文件命名规范：case.yaml
  - 脚本文件命名规范：与目录同名
  - 支持多语言脚本文件
  - **强制要求**：配置文件和脚本文件必须放在同一个目录下

### 3.12 KB脚本编写逻辑
- **功能描述**：定义KB脚本的编写规范和执行逻辑
- **输入**：脚本文件、配置文件
- **输出**：标准化的执行结果
- **详细需求**：
  - **编写模式**：采用"一个脚本一个配置文件"的模式
  - **目录结构**：每个CASE必须有独立的目录，包含case.yaml和对应的脚本文件
  - **配置模板**：case.yaml必须包含以下字段：
    - name：CASE名称
    - language：脚本语言（bash/python）
    - category：分类
    - tags：标签列表
    - description：描述
    - timeout：超时时间
    - weight：权重
    - params：业务配置参数
  - **脚本模板**：
    - 初始化必要的环境变量
    - 提供标准的API调用框架
    - 支持步骤化执行和结果记录
    - 统一的错误处理机制

### 3.13 KB脚本加载机制
- **功能描述**：执行器加载kbscript目录下的CASE
- **输入**：kbscript目录路径
- **输出**：加载的CASE列表
- **详细需求**：
  - 自动扫描kbscript目录及其子目录
  - 识别包含case.yaml配置文件的目录作为CASE
  - 支持按分类、标签筛选加载的CASE
  - 验证CASE配置的完整性和正确性
  - 缓存加载结果以提高性能

### 3.14 Web服务模块（P1）
- **功能描述**：提供HTTP服务和内嵌Web前端
- **输入**：HTTP请求
- **输出**：HTTP响应、静态资源
- **详细需求**：
  - 提供 `serve` 命令启动HTTP服务
  - 支持指定监听地址和端口
  - 支持Token认证
  - 内嵌Web前端静态资源
  - 提供RESTful API接口
  - 支持CORS跨域请求
  - CLI与Web服务功能隔离，互不影响

### 3.15 Token认证模块（P1）
- **功能描述**：提供Token认证保护Web服务
- **输入**：Token验证请求
- **输出**：验证结果、会话Cookie
- **详细需求**：
  - 支持通过命令行参数 `--token` 指定Token
  - 支持通过配置文件 `server.token` 指定Token
  - Token可选，不配置时不需要认证（仅本地使用）
  - Token验证成功后设置会话Cookie
  - 未认证用户访问时重定向到登录页面
  - 登录页面简洁，只需输入Token
  - 支持API Token验证（Header Authorization: Bearer token）
  - 启动时显示Token信息（Raw Token和Hashed Token）

### 3.15.1 Token类型与角色
- **开发者Token（固定）**：`AdminDEFA123`
  - 角色：admin
  - 权限：可编辑SKILL.md、可查看版本历史、可回滚
- **用户Token（随机）**：配置文件指定
  - 角色：user
  - 权限：仅查看执行结果和详情

### 3.15.2 角色识别与权限
- 登录时识别Token类型，设置 `kb_role` Cookie
- 管理员Token：`kb_role=admin`
- 用户Token：`kb_role=user`
- 前端通过 `IsAdmin()` 判断当前用户角色

### 3.16 执行历史模块（P1）
- **功能描述**：记录和管理执行历史
- **输入**：执行结果
- **输出**：历史记录列表、历史详情
- **详细需求**：
  - 每次执行生成一条历史记录
  - 历史记录存储在本地文件（JSON格式）
  - 支持查询历史记录列表
  - 支持查询单条历史详情
  - 支持删除历史记录
  - 历史记录文件命名：`history_{execution_id}.json`
  - 支持自动清除：当历史记录超过配置的最大条数时，自动删除最旧的记录
  - 默认最大条数：4294967296（2^32）

### 3.21 ICare日志适配器模块
- **功能描述**：提供从ICare日志目录结构中获取日志的能力，技术支持人员只需提供Q单号
- **输入**：Q单号、日志文件路径、搜索关键字
- **输出**：日志文件列表、日志内容、搜索结果
- **详细需求**：

#### 3.21.1 日志目录结构
日志根目录：`/sf/data/icare_log/logall`
```
logall/
├── {YYMM}/              # 年月份目录（如2603表示2026年3月）
│   ├── Q{yymmddnnnn}/   # Q单号目录（如Q2026031700424）
│   │   ├── _members
│   │   ├── cfgmaster_ini
│   │   ├── collect_record.txt
│   │   ├── log_list.json
│   │   ├── Q2026031700424.zip  # 可选的压缩包
│   │   └── {host}/      # 主机目录（如sp_10.250.0.7）
│   │       ├── alert/
│   │       ├── auth/
│   │       ├── boot/
│   │       └── ... (各种日志文件)
│   └── ...
└── ...
```

#### 3.21.2 功能需求
| 功能 | 描述 |
|------|------|
| 自动解析Q单号 | 从Q单号自动解析年月份（如Q2026031700424→2603） |
| 自动查找主机 | 自动查找Q单号下的所有主机目录 |
| 自动解压 | 检测ZIP文件并自动解压 |
| 文件列表 | 获取指定条件下的所有日志文件 |
| 内容读取 | 读取指定日志文件的完整内容 |
| 关键字搜索 | 在日志文件中搜索指定关键字 |

#### 3.21.3 简化API设计原则
- 技术支持人员只需提供Q单号
- 其他信息（年月份、主机）由系统自动推断
- 如存在压缩包自动解压后再读取

#### 3.21.4 Bash API设计
```bash
# 一键初始化（只需Q单号）
source scripts/bash/icare_log_api.sh
icare Q2026031700424

# 或分步操作
icare_log_init Q2026031700424      # 只需Q单号
icare_log_set_host sp_10.250.0.7   # 切换主机（可选）

# 获取主机列表
icare_log_list_hosts

# 获取日志文件列表（返回JSON数组）
icare_log_list

# 读取日志内容
icare_log_read "alert/alert.log"

# 搜索日志
icare_log_search "error"
```

#### 3.21.5 Python API设计
```python
# 一键初始化（只需Q单号）
from icare_log_api import icare
adapter = icare("Q2026031700424")

# 或分步操作
from icare_log_api import IcareLogAdapter
adapter = IcareLogAdapter()
adapter.init("Q2026031700424")      # 只需Q单号
adapter.set_host("sp_10.250.0.7")   # 切换主机（可选）
icare_log.set_host("sp_10.250.0.7")

# 获取文件列表
files = icare_log.list()

# 读取日志
content = icare_log.read("alert.log")

# 搜索
results = icare_log.search("error")

# 时间范围搜索
content = icare_log.read_range("alert.log", "2025-03-01", "2025-03-10")

# 获取列表
yearmonths = icare_log.list_yearmonths()
qnos = icare_log.list_qnos()
hosts = icare_log.list_hosts()
```

#### 3.21.5 错误处理
- 日志目录不存在时返回友好错误提示
- 日志文件不存在时返回错误信息
- 搜索无结果时返回空结果集
- 文件读取失败时返回错误信息

### 3.22 智能日志适配器模块（上传分析场景）

- **功能描述**：针对客户上传日志进行分析的场景，智能日志适配器自动识别日志类型、自动解压压缩文件、提供统一的查询接口
- **输入**：Q单号、查询条件（关键字、时间范围、SQL语句）
- **输出**：黑盒子日志内容、告警/操作日志查询结果、监控指标数据
- **详细需求**：

#### 3.22.1 日志目录结构（上传分析场景）

#### 3.22.2 日志类型自动识别
| 类型 | 识别路径 | 说明 |
|------|----------|------|
| 黑盒子日志 | `log/blackbox/{日期}/LOG_*.txt` | 系统状态快照，包含cpu/内存/磁盘/网络等详细信息 |
| 数据库日志 | `log/log_new.db` | SQLite数据库，包含alert表(告警)和log表(操作日志) |

| 监控指标 | `log/checkitem/*_date` | 性能监控数据文件（cpu_rate、mem_ratio等） |
| 应用日志 | `log/{name}/` | 各应用日志目录（host-agentd、apache2等） |

#### 3.22.3 数据库日志结构（log_new.db）

  - 字段：id, type, host, hostname, vm, user, description, start, end, status, process等

- **vmwarealert表**：VMware告警

#### 4.5.1 日志级别
| 级别 | 用途 | 示例 |
|------|------|------|

### 4.6 结果输出规范需求

系统应提供标准化的结果输出格式，确保结果的一致性和可读性。

#### 4.6.1 步骤结果规范

每个步骤的执行结果应包含以下字段：

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| name | string | 是 | 步骤名称 |
| status | enum | 是 | 状态：success/failure/warning/skipped |
| message | string | 否 | 执行消息 |
| output | string | 否 | 原始输出 |
| duration_ms | number | 否 | 执行耗时(毫秒) |

#### 4.6.2 脚本结果规范

脚本的整体执行结果应包含以下字段：

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| script_name | string | 是 | 脚本名称 |
| status | enum | 是 | 整体状态 |
| steps | array | 是 | 步骤结果列表 |
| score | number | 否 | 执行得分(0-1) |
| duration_ms | number | 否 | 总耗时(毫秒) |
| message | string | 否 | 整体消息 |

#### 4.6.3 结果API设计

**Bash API**：
```bash

step_skip "message"              # 跳过

# 记录键值对结果

```

**Python API**：

```yaml
  cleanup_threshold: 0.9           # 清理阈值（达到90%时触发）
```

### 11.6 清除策略

- **自动清除**：当历史记录数达到 `max_records * cleanup_threshold` 时，自动删除最旧的记录
- **手动清除**：
  - API: `DELETE /api/v1/history` 清除全部
  - API: `DELETE /api/v1/history/{id}` 清除单条
  - CLI: `kb-runner history clear` 清除全部

## 14. 风险/依赖备注

### 10.1 风险
- **脚本执行安全**：脚本可能包含恶意代码，需要加强执行隔离
- **API一致性**：不同脚本语言的API实现可能存在差异，需要统一接口设计
- **性能瓶颈**：并行执行多个脚本可能导致系统资源不足，需要优化资源管理
- **结果准确性**：权重配置的合理性可能影响结果的准确性，需要提供配置指导

### 10.2 依赖
- **脚本语言环境**：依赖bash、Python等脚本语言的运行环境
- **Go依赖库**：依赖必要的Go第三方库
- **后端系统**：依赖后端系统的API进行结果导入
- **操作系统**：依赖操作系统提供的进程管理和文件系统功能
