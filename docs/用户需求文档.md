# KB脚本执行框架用户需求文档

## 1. 简介

### 1.1 文档目的
本文档描述KB脚本执行框架的用户需求，旨在为开发团队提供清晰的功能需求。

### 1.2 术语定义
| 术语 | 解释 |
|------|------|
| KB | Knowledge Base，知识库，包含检查逻辑和规则 |
| CASE | 一个可执行的检查脚本，包含脚本路径、参数、权重等配置 |
| 场景 | 一组CASE的集合，代表一个完整的检查流程 |

## 2. 用户使用方式

### 2.1 使用场景概述

KB脚本执行框架面向两类用户群体：

| 优先级 | 使用方式 | 目标用户 | 特点 |
|--------|----------|----------|------|
| P0 | CLI工具 | 运维人员、开发人员 | 命令行直接执行，无需Web界面 |
| P1 | Web界面 | 测试人员、管理人员 | 图形化操作，依赖CLI核心执行器 |

### 2.2 CLI工具核心需求（P0）

#### 2.2.1 查看CASE列表

**用户需求**：用户需要查看当前有哪些可用的CASE。

**命令示例**：
```bash
kb-runner list                    # 列出所有CASE
kb-runner list --category security # 按分类筛选
kb-runner list --tags critical    # 按标签筛选
kb-runner show <case-name>        # 查看CASE详情
```

#### 2.2.2 选择场景执行

**用户需求**：用户需要选择预定义的场景来执行一组CASE。

**命令示例**：
```bash
kb-runner scenario list           # 列出所有场景
kb-runner scenario show <name>    # 查看场景详情
kb-runner run --scenario daily_check  # 按场景执行
```

#### 2.2.3 交互式选择CASE

**用户需求**：用户需要通过交互式界面选择要执行的CASE。

**命令示例**：
```bash
kb-runner run --interactive       # 交互式选择CASE执行
kb-runner run -i                  # 简写形式
```

#### 2.2.4 执行脚本

**用户需求**：用户需要执行脚本并获取结果。

**命令示例**：
```bash
kb-runner run -s ./scripts/check.sh -l bash  # 执行单个脚本
kb-runner run --case security_check          # 按CASE名称执行
kb-runner run --scenario daily_check         # 按场景执行
```

### 2.3 Web界面核心需求（P1）

#### 2.3.1 启动Web服务

**用户需求**：用户需要启动Web服务，通过浏览器访问图形化界面。

**命令示例**：
```bash
kb-runner serve                    # 启动Web服务（默认端口8080）
kb-runner serve --port 9090        # 指定端口启动
kb-runner serve --host 0.0.0.0     # 指定监听地址
kb-runner serve --token mytoken     # 指定访问Token
```

**使用场景**：
- 测试人员通过浏览器访问 http://localhost:8080 进行操作
- 交付物为单个二进制文件，无需额外部署前端
- 服务暴露到公网时，需要Token验证才能访问

#### 2.3.2 Token认证

**用户需求**：当服务暴露到公网时，需要Token验证防止未授权访问。

| 功能 | 描述 |
|------|------|
| Token验证 | 首次访问需输入Token验证 |
| 登录页 | 简洁的登录界面 |
| 会话保持 | 验证成功后保持登录状态 |
| 配置Token | 支持配置文件或命令行指定Token |

#### 2.3.2 CASE管理界面

**用户需求**：通过Web界面查看和管理CASE。

| 功能 | 描述 |
|------|------|
| CASE列表 | 以表格形式展示所有CASE，支持按分类、标签筛选 |
| CASE详情 | 点击CASE查看详细配置信息 |
| CASE搜索 | 按名称或描述搜索CASE |

#### 2.3.3 场景管理界面

**用户需求**：通过Web界面查看和管理场景。

| 功能 | 描述 |
|------|------|
| 场景列表 | 展示所有预定义场景 |
| 场景详情 | 查看场景包含的CASE列表和执行配置 |
| 场景执行 | 一键执行场景中的所有CASE |

#### 2.3.4 执行管理界面

**用户需求**：通过Web界面执行脚本并配置参数。

| 功能 | 描述 |
|------|------|
| 选择执行 | 选择单个或多个CASE执行 |
| 参数配置 | 执行前配置脚本参数 |
| 执行配置 | 配置超时时间、并行数等 |
| 实时状态 | 查看执行进度和状态 |

#### 2.3.5 结果查看界面

**用户需求**：通过Web界面查看执行结果和历史记录。

| 功能 | 描述 |
|------|------|

## 3. 功能需求

### 3.1 CASE管理

- **需求描述**：支持查看当前有哪些CASE，支持按分类、标签筛选
- **用户场景**：用户想知道系统中有哪些可执行的检查脚本

### 3.2 场景管理

- **需求描述**：支持查看预定义的场景，选择场景执行
- **用户场景**：用户想执行"日常巡检"场景，一次性运行多个相关CASE

### 3.3 脚本执行

- **需求描述**：支持执行脚本并获取标准化结果
- **用户场景**：用户执行检查脚本，获取执行状态和结果

### 3.4 结果处理

- **需求描述**：支持结果加权、归一化、矩阵化
- **用户场景**：用户需要查看多个脚本执行的综合评分

### 3.5 日志规范

- **需求描述**：脚本执行过程中的日志输出需要规范，便于排查问题
- **用户场景**：用户需要通过日志快速定位脚本执行过程中的问题

### 3.6 结果输出规范

- **需求描述**：脚本执行结果需要统一格式，步骤返回结果需要规范
- **用户场景**：用户需要查看标准化的执行结果，便于分析和对比

### 3.7 CASE目录结构

- **需求描述**：技术支持编写KB脚本时，需要采用"一个脚本一个配置文件"的模式，配置文件和脚本必须放在同一个路径下，每个脚本一个独立目录
- **用户场景**：技术支持需要快速创建标准化的CASE目录结构，包含脚本文件和配置文件
- **命令示例**：
```bash
kb-runner init case_name       # 创建CASE目录模板
kb-runner init --name security_check --lang bash  # 指定语言创建
```

### 3.8 CASE目录模板

- **需求描述**：CASE目录需要包含标准化的文件结构，便于管理和维护
- **用户场景**：技术支持需要按照统一模板创建CASE，保证配置一致性
- **模板要求**：
  - 配置模板：定义必需的字段（name、language、category、tags、description、timeout、weight、params）
  - 脚本模板：初始化必要的环境变量，提供标准的API调用框架
  - KB业务配置：可在配置文件的params字段中定义业务特定的配置项

### 3.9 KB脚本加载

- **需求描述**：执行器可以加载kbscript目录下存放的CASE
- **用户场景**：技术支持将编写好的CASE存放在kbscript目录，执行器能够自动识别和加载
- **加载逻辑**：
  - 自动扫描kbscript目录及其子目录
  - 识别包含case.yaml配置文件的目录作为CASE
  - 支持按分类、标签筛选加载的CASE

### 3.9 Web服务（P1）

- **需求描述**：提供内嵌Web服务，用户通过浏览器访问图形化界面
- **用户场景**：测试人员启动服务后，通过浏览器进行CASE执行和结果查看
- **交付要求**：单个二进制文件，内嵌前端资源，无需额外部署

### 3.10 执行历史（P1）

- **需求描述**：记录执行历史，支持查看和导出
- **用户场景**：用户需要回顾历史执行记录，导出结果报告
- **存储要求**：历史记录存储在本地文件，结果格式符合标准化定义

### 3.11 结果导出（P1）

- **需求描述**：支持将执行结果导出为标准格式
- **用户场景**：用户需要将执行结果导出后存档或分享给他人
- **支持格式**：JSON、YAML

### 3.12 Token认证（P1）

- **需求描述**：Web服务暴露到公网时，需要Token验证防止未授权访问
- **用户场景**：用户将服务部署到公网时，需要Token保护
- **功能要求**：
  - 支持命令行参数指定Token
  - 支持配置文件指定Token
  - 未授权访问时显示登录页面
  - Token验证成功后保持会话

### 3.13 执行结果柱状图展示（P1）

**需求描述**：CTI客户在执行完选择的CASE后，显示一个按得分排序的柱状图。

**详细说明**：
- 柱状图 X 轴为 CASE 名称（或 KB 编码）
- 柱状图 Y 轴为归一化分数（0-100%）
- 柱状图按照得分从高到低排序
- 支持显示所有执行的 CASE 的得分对比
- 柱状图应采用响应式设计，支持移动端查看

**验收标准**：
- 执行完成后自动显示柱状图
- 柱状图数据与实际执行结果一致
- 支持按归一化分数排序

### 3.14 柱状图交互 - 鼠标悬停显示KB编码（P1）

**需求描述**：鼠标悬停在柱子上时，显示该 CASE 对应的 KB 编码。

**详细说明**：
- 悬停时显示 Tooltip，内容包括：KB 编码、CASE 名称、得分
- Tooltip 样式简洁，与整体界面风格一致
- 响应时间 < 100ms

**验收标准**：
- 鼠标悬停时正确显示 KB 编码
- 鼠标移开后 Tooltip 正确隐藏

### 3.15 柱状图交互 - 点击查看执行详情（P1）

**需求描述**：点击柱子可以查看当前 CASE 的执行详情。

**详细说明**：
- 点击后打开详情弹窗或跳转至详情页面
- 详情内容应包含：执行状态、得分、权重、每一步骤的匹配情况

**验收标准**：
- 点击柱子能够打开执行详情页面
- 详情页面包含完整的执行信息

### 3.16 执行详情展示（P1）

**需求描述**：执行详情页展示 CASE 执行的所有关键信息。

**详细说明**：

详情页面需包含以下内容：

1. **基本信息**
   - CASE 名称
   - KB 编码
   - 执行时间
   - 执行状态（成功/失败/警告）
   - 归一化得分

2. **配置信息**
   - KB 权重
   - 要分析的对象列表
   - 执行参数

3. **Skill.md 内容**
   - 渲染后的 Skill.md 文件完整内容
   - 高亮显示匹配成功的规则
   - 显示未匹配的规则

4. **执行日志**
   - KB 脚本调用 API 打印的日志
   - 每一步的匹配情况
   - 错误信息（如有）

5. **步骤详情**
   - 每个检查步骤的名称
   - 步骤状态（success/failure/warning）
   - 步骤耗时
   - 步骤输出结果

**验收标准**：
- 详情页显示完整的基本信息
- 详情页显示 KB 配置信息
- 详情页显示 Skill.md 内容和匹配情况
- 详情页显示完整的执行日志

### 3.18 新标签页显示Skill.md（P1）

- 在CASE列表页面，点击"详情"按钮时，在新标签页中打开Skill.md渲染内容

- 新标签页独立存在，切换返回后不会产生遮罩残留问题
- 页面包含CASE基本信息（KB编码、分类、标签）

**验收标准**：
- 点击详情按钮在新标签页打开
- Markdown内容正确渲染
- 切换标签页后无遮罩残留

### 3.17 Skill.md 编辑功能（P1）

- 技术人员可以编辑并保存 Skill.md

- 保存后可重新执行验证
- 版本历史能够查看和回滚

### 3.18 ICare日志适配器

**需求描述**：KB脚本经常需要分析ICare系统的日志信息，日志适配器提供从指定目录结构中获取日志的能力。

**详细说明**：
日志目录结构为：`/sf/data/icare_log/logall/{年月份}/{Q单号}/{主机}/{日志文件}`
- 技术支持人员只需提供Q单号（如Q2026031700424）
- 系统自动从Q单号解析年月份（Q2026031700424 → 2603）
- 自动查找该Q单号下的所有主机
- 如日志为ZIP压缩包，自动解压

**功能需求**：
| 功能 | 描述 |
|------|------|
| 自动解析 | 从Q单号自动解析年月份 |
| 自动查找主机 | 自动查找Q单号下的所有主机 |
| 自动解压 | 如存在ZIP文件自动解压 |
| 获取文件列表 | 列出指定条件下的所有日志文件 |
| 读取日志内容 | 读取指定日志文件的内容 |
| 关键字搜索 | 支持在日志中搜索关键字 |

**命令示例（Bash）**：
```bash
# 一键初始化（只需Q单号）
source scripts/bash/icare_log_api.sh
icare Q2026031700424

# 或分步操作
icare_log_init Q2026031700424      # 只需Q单号，自动解析年月份、查找主机
icare_log_set_host sp_10.250.0.7   # 切换主机（可选）

# 获取日志文件列表
icare_log_list

# 读取日志内容
icare_log_read "alert/alert.log"

# 搜索日志
icare_log_search "error"
```

**命令示例（Python）**：
```python
import sys
sys.path.insert(0, 'scripts/python')
from icare_log_api import *

# 一键初始化（只需Q单号）
adapter = icare("Q2026031700424")

# 或分步操作
adapter.init("Q2026031700424")     # 只需Q单号
adapter.set_host("sp_10.250.0.7")  # 切换主机（可选）

# 获取文件列表
files = adapter.list()

# 读取日志
content = adapter.read("alert.log")

# 搜索
results = adapter.search("error")
```

**验收标准**：
- 只需提供Q单号即可获取日志
- 自动从Q单号解析年月份
- 自动查找并列出所有主机
- 自动解压ZIP文件（如有）
- 能够列出日志文件列表
- 能够读取日志文件内容
- 能够搜索日志关键字

### 3.19 智能日志适配器（上传分析场景）

**需求描述**：当客户需要分析某个虚拟机的问题时，技术支持上传日志后，智能日志适配器需要自动识别日志类型、自动解压压缩文件、并提供统一的查询接口。

**使用场景**：
1. 客户提供Q单号，系统自动定位日志目录
2. 自动识别日志类型：黑盒子日志、告警/操作日志、系统日志、监控指标
3. 自动解压压缩文件（tar.gz, zip）
4. 支持数据库日志的SQL查询、关键字搜索、时间范围查询
5. 技术支持只需关注Q单号，其他自动处理

**详细说明**：

#### 日志目录结构
```
/sf/data/icare_log/logall/{年月份}/{Q单号}/{主机}/
├── log/
│   ├── 15.tar.gz                          # 系统日志（需解压）
│   ├── log_new.db                         # 数据库日志（告警/操作）
│   ├── blackbox/                          # 黑盒子日志
│   │   └── {日期}/                        # 如 20260315/
│   │       ├── LOG_*.txt                  # 各种黑盒子日志
│   │       └── LOG_*.txt.zip              # 压缩的黑盒子日志（需解压）
│   ├── checkitem/                         # 监控指标数据
│   │   ├── cpu_rate_date
│   │   ├── mem_ratio_date
│   │   └── ...
│   ├── host-agentd/                       # 主机代理日志
│   ├── apache2/                           # Apache日志
│   └── ...
├── alert/                                 # 告警配置
└── ...
```

#### 日志类型识别
| 类型 | 识别方式 | 说明 |
|------|----------|------|
| 黑盒子日志 | `log/blackbox/{日期}/LOG_*.txt` | 系统状态快照，包含cpu、内存、磁盘、网络等 |
| 数据库日志 | `log/log_new.db` | SQLite数据库，包含alert表和log表 |
| 系统日志 | `log/15.tar.gz` | 系统日志归档，需解压后读取 |
| 监控指标 | `log/checkitem/*_date` | 性能监控数据文件 |

#### 数据库日志结构（log_new.db）
- **alert表**：告警记录（13136条），包含type、host、object_type、object_name、description、start、level等字段
- **log表**：操作日志（2875条），包含type、host、user、description、start、end、status等字段

#### 功能需求
| 功能 | 描述 |
|------|------|
| 自动识别 | 自动识别日志目录下的所有日志类型 |
| 自动解压 | 自动检测并解压tar.gz和zip文件 |
| 黑盒子查询 | 提供黑盒子日志的关键字搜索和内容读取 |
| 数据库查询 | 支持SQL查询、关键字搜索、时间范围筛选 |
| 监控数据 | 提供监控指标数据的读取接口 |
| 统一入口 | 只需Q单号，其他自动处理 |

#### 命令示例（Bash）
```bash
# 一键初始化（只需Q单号）
source scripts/bash/smart_log_api.sh
smart_icare Q2026031700281

# 初始化后自动识别所有日志类型
smart_icare_init Q2026031700281

# 查询黑盒子日志
smart_log_search "error"                    # 搜索关键字
smart_log_read "blackbox/20260315/LOG_dmesg.txt"  # 读取黑盒子日志

# 查询数据库日志
smart_db_query "SELECT * FROM alert LIMIT 10"     # SQL查询
smart_db_search "disk"                             # 搜索告警/日志
smart_db_alerts --level critical                   # 查询严重告警
smart_db_logs --hours 24                           # 查询24小时内的操作日志

# 查询监控指标
smart_metric cpu_rate                              # CPU使用率
smart_metric mem_ratio                             # 内存使用率

# 状态查看
smart_log_status                                   # 显示当前日志适配器状态
```

#### 命令示例（Python）
```python
from smart_log_api import smart_icare

# 一键初始化
adapter = smart_icare("Q2026031700281")

# 黑盒子日志
adapter.blackbox.search("error")
```
